1. System Overview & RBAC Architecture

JENE-DFCMS implements a Role-Based Access Control (RBAC) system with 6 defined roles. Access control is enforced at two levels:

  • Server-side (Spring Boot controllers): Each API endpoint checks the authenticated user's role via RoleHelper before processing. Unauthorized requests receive HTTP 403 with descriptive error messages.
  • Client-side (Angular frontend): UI elements are conditionally displayed/hidden based on the logged-in user's roles. Navigation, buttons, and forms adapt per role.
Key Principle: Server-side enforcement is the primary security layer. The frontend hides elements but cannot be relied upon for access control. All API requests require a valid JWT token, and role checks happen server-side in every controller method.

Authentication Flow

  1. User submits credentials to POST /api/auth/login
  2. Server validates credentials against BCrypt-hashed password
  3. Server generates JWT token containing username and roles
  4. Frontend stores token in localStorage and attaches it to all subsequent requests via Authorization: Bearer <token> header
  5. On each request, JwtAuthFilter extracts username, loads user from DB with roles
  6. RoleHelper in each controller checks if the user has the required role for the specific operation

Role Data Model

Roles are stored in the roles table and linked to users via user_roles. A user can have multiple roles. Example:

user_roles(user_id=1, role_id=1) → User 1 has ADMIN
user_roles(user_id=2, role_id=3) → User 2 has INVESTIGATOR
user_roles(user_id=3, role_id=2) → User 3 has CASE_MANAGER

2. Role Definitions

ADMIN Full System Administrator

The highest-privilege role. Has unrestricted access to all system features including user management, case management, task management, evidence management, and reporting. Can authorize cases, manage system settings, and perform all operations. Typically assigned to system administrators and senior forensic managers.

  • Can create, edit, delete, and manage all users
  • Can perform all case, task, evidence, and report operations
  • Can authorize cases (approve/reject)
  • Can change system-wide settings
  • Can view all data regardless of assignment
CASE_MANAGER Case Management Lead

Responsible for managing the lifecycle of forensic cases. Can create and edit cases, assign tasks to investigators and QA staff, manage case status, and link/unlink related cases. Cannot authorize cases or manage users.

  • Can create and edit cases
  • Can assign tasks to investigators and QA staff
  • Can manage case status (except authorization)
  • Can link/unlink related cases
  • Can create tasks and evidence entries
  • Cannot authorize cases or manage users
INVESTIGATOR Forensic Investigator

Works on assigned forensic tasks. Can create and manage evidence, add forensic notes, check-in/check-out evidence items, and update task status. Can only work on tasks/cases they are assigned to.

  • Can view assigned cases and tasks
  • Can create and edit evidence
  • Can check-in/check-out evidence (chain of custody)
  • Can add forensic notes to tasks
  • Can update status of assigned tasks
  • Cannot create cases or tasks, cannot assign work
QA Quality Assurance Reviewer

Reviews forensic work for quality and compliance. Can view tasks and add notes for review purposes. Has read-only access to evidence. Focuses on ensuring work meets forensic standards and legal requirements.

  • Can view assigned cases and tasks
  • Can add review notes to tasks
  • Can view evidence (read-only)
  • Cannot create or modify evidence, cases, or tasks
  • Cannot check-in/check-out evidence
REQUESTER Case Requester

Submits new forensic case requests. Can create new cases but cannot manage tasks, evidence, or other aspects of case execution. Typically used by law enforcement officers or legal teams requesting forensic analysis.

  • Can create new cases (submit requests)
  • Can view their own submitted cases
  • Cannot create tasks or evidence
  • Cannot modify existing cases beyond initial submission
AUTHORISER Case Authoriser / Approver

Has authority to approve or reject case authorization requests. This is the final sign-off role in the forensic case lifecycle. Can also close cases. Typically assigned to senior officials or legal authorities.

  • Can authorize (approve/reject) cases
  • Can close cases
  • Can change case status
  • Cannot create cases, tasks, or evidence
  • Cannot manage users

4. Case Management Permissions (Detailed)

Case Lifecycle States

A case progresses through the following states. Role-based transitions are enforced:

StateWho Can TransitionNext State(s)
DRAFTCASE_MANAGER, ADMINOPEN, CLOSED
OPENCASE_MANAGER, ADMININ_PROGRESS, CLOSED
IN_PROGRESSCASE_MANAGER, ADMINPENDING_AUTH, CLOSED
PENDING_AUTHAUTHORISER, ADMINAUTHORIZED, REJECTED
AUTHORIZEDCASE_MANAGER, ADMINCLOSED
REJECTEDAUTHORISER, ADMINOPEN, CLOSED
CLOSEDCASE_MANAGER, ADMIN(terminal)

Case Operations Detail

  • Create Case: ADMIN, CASE_MANAGER, REQUESTER — Creates a new forensic case with title, description, priority, classification, type
  • Edit Case: ADMIN, CASE_MANAGER — Modifies case details (title, description, priority, classification, type, assigned team)
  • Change Status: ADMIN, CASE_MANAGER, AUTHORISER — Transitions case through lifecycle states with required reason field
  • Close Case: ADMIN, CASE_MANAGER, AUTHORISER — Moves case to terminal CLOSED state with reason
  • Authorize Case: ADMIN, AUTHORISER only — Approves or rejects case authorization request with reason
  • Link Cases: ADMIN, CASE_MANAGER — Links two related cases for cross-referencing
  • View History: All authenticated users (filtered by assignment)

Task Assignment

  • Assign Investigator: ADMIN, CASE_MANAGER — Assigns primary or secondary investigator to task
  • Assign QA: ADMIN, CASE_MANAGER — Assigns primary or secondary QA reviewer to task
  • Investigators/QA cannot self-assign — Assignment is always done by Case Manager or Admin

Forensic Notes

  • Add Notes: ADMIN, CASE_MANAGER, assigned INVESTIGATOR, assigned QA
  • View Notes: All users with task access
  • Author tracking: Notes are timestamped and linked to the author for audit purposes

6. Evidence Management Permissions (Detailed)

Evidence Lifecycle States

StateWho Can TransitionNext State(s)
COLLECTEDADMIN, CASE_MANAGER, assigned INVESTIGATORIN_ANALYSIS, IN_STORAGE
IN_ANALYSISADMIN, CASE_MANAGER, assigned INVESTIGATORCOLLECTED, IN_STORAGE, ANALYZED
IN_STORAGEADMIN, CASE_MANAGER, assigned INVESTIGATORCOLLECTED, IN_ANALYSIS
ANALYZEDADMIN, CASE_MANAGERCOLLECTED, IN_STORAGE

Chain of Custody

  • Check-out: ADMIN, CASE_MANAGER, assigned INVESTIGATOR — Records evidence removal from storage with custodian name, date, and comments
  • Check-in: ADMIN, CASE_MANAGER, assigned INVESTIGATOR — Records evidence return to storage with custodian name, date, and comments
  • View Chain: All users with evidence access (QA has read-only)
  • Chain of Custody is immutable — Entries cannot be modified or deleted once created

Evidence Types Supported

  • Digital: Hard drives, USB drives, memory cards, network captures
  • Physical: Documents, weapons, biological samples
  • Each evidence item has: title, description, type, status, collected date, location, hash value

8. Reports & Dashboard Permissions

Reporting and dashboard access is available to all authenticated users. The dashboard shows real-time statistics including:

  • Total cases and their status distribution
  • Active tasks and completion rates
  • Evidence items and their status
  • Recent activity across the system
Note: While all users can access the dashboard and reports, the underlying data is filtered based on each user's assigned cases and tasks. For example, an INVESTIGATOR only sees statistics for their assigned work.

9. Forensic Notes Permissions

Forensic notes are critical for maintaining an audit trail of investigative work:

  • Create note: ADMIN, CASE_MANAGER, assigned INVESTIGATOR, assigned QA
  • View notes: All users with access to the parent task
  • Edit/Delete notes: NOT ALLOWED (notes are immutable once created)
  • Author tracking: Each note records the author (user ID) and creation timestamp
  • Case history: All case-level actions (status changes, assignments, etc.) are automatically recorded as history entries

10. Authorization Flow

Case Authorization Workflow

  1. CASE_MANAGER creates case → status: DRAFT
  2. CASE_MANAGER opens case → status: OPEN
  3. CASE_MANAGER starts work → status: IN_PROGRESS
  4. CASE_MANAGER requests authorization → status: PENDING_AUTH
  5. AUTHORISER reviews and approves → status: AUTHORIZED
  6. CASE_MANAGER closes case → status: CLOSED

Rejection Flow

  1. Case is at PENDING_AUTH
  2. AUTHORISER rejects with reason → status: REJECTED
  3. CASE_MANAGER or ADMIN reopens → status: OPEN
  4. Work continues and new authorization is requested

Evidence Chain of Custody Flow

  1. INVESTIGATOR collects evidence → status: COLLECTED
  2. INVESTIGATOR checks out evidence → records custodian, date, comment
  3. INVESTIGATOR analyzes evidence → status: IN_ANALYSIS
  4. INVESTIGATOR checks in evidence → records return to storage
  5. Each check-in/check-out creates an immutable chain of custody entry

12. Server-Side Enforcement Details

RoleHelper Component

All role checks are centralized in com.foreman.config.RoleHelper:

  • Extracts the current user from the JWT token on each request
  • Loads user from database with roles (EAGER fetch)
  • Provides convenience methods: isAdmin(), isCaseManager(), isInvestigator(), isQA(), isRequester(), isAuthoriser()
  • Provides composite methods: canManageCases(), canCreateCases(), canAuthorizeCases(), canManageTasks(), canWorkOnTasks(), canManageEvidence(), canManageUsers()

Error Handling

When a user attempts an unauthorized action, the system returns:

  • HTTP 403 Forbidden with message: "Access denied: [specific reason]"
  • HTTP 401 Unauthorized when no valid JWT token is provided
  • Frontend displays error toasts with the specific denial reason

Enforcement Points

ControllerEnforcement
CaseControllerRoleHelper injected; requireAuth() on all endpoints; canCreateCases(), canManageCases(), canAuthorizeCases(), canManageTasks(), canManageEvidence() on mutations
TaskControllerRoleHelper injected; requireAuth() on all endpoints; canManageTasks() on create/update/assign; canWorkOnTasks() on status change and notes
EvidenceControllerRoleHelper injected; requireAuth() on all endpoints; canManageEvidence() on create/update/status/check-in/check-out
UserControllerRoleHelper injected; requireAuth() on all endpoints; requireAdmin() on create/update
ReportControllerDashboard and reports accessible to all authenticated users
