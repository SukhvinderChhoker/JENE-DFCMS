4. Critical Findings (Immediate Action Required)

CRITICAL

C1: JWT Token Signature Not Verified - Full Authentication Bypass

OWASP: A08:2021 - Data Integrity Failures

CWE: CWE-345 (Insufficient Verification of Data Authenticity)

File: RoleHelper.java:29-36

The extractUsernameFromToken() method in RoleHelper manually decodes the JWT payload using Base64 without verifying the HMAC signature. This means any attacker can craft a JWT token with any username (including admin) and the system will accept it as valid.

String[] parts = token.split("\\."); String payload = new String(java.util.Base64.getUrlDecoder().decode(parts[1])); // NO SIGNATURE VERIFICATION

Impact: Complete authentication bypass. Attacker can impersonate any user including administrators. Full system compromise.

Remediation: Use JwtUtil.extractUsername() which already performs signature verification. Replace manual Base64 parsing with the existing JWT library validation. Never parse JWTs manually.
CRITICAL

C2: Hardcoded JWT Secret in Source Control

OWASP: A02:2021 - Cryptographic Failures

CWE: CWE-798 (Use of Hard-coded Credentials)

File: application.properties:15

The JWT signing secret jwt.secret=404E635266556A586E3272357538782F413F4428472B4B6250645367566B5970 is hardcoded and committed to the public GitHub repository. Anyone with access can forge valid JWT tokens.

Impact: Anyone with repository access can forge tokens for any user, bypassing all authentication and authorization.

Remediation: Move to environment variable: jwt.secret=${JWT_SECRET}. Generate a new 256-bit random secret. Rotate immediately. Use a secrets manager (Vault, AWS SSM) in production.
CRITICAL

C3: H2 Database Console Exposed - Unauthenticated DB Access

OWASP: A05:2021 - Security Misconfiguration

CWE: CWE-200 (Exposure of Sensitive Information)

Files: application.properties:12-13, SecurityConfig.java:39,45

The H2 console is enabled, accessible at /h2-console without authentication, and framed with sameOrigin headers. The database has an empty password. This provides direct SQL access to the entire database including user credentials.

Impact: Full unauthenticated database takeover. Read/drop all data including passwords, cases, evidence.

Remediation: Disable in production: spring.h2.console.enabled=false. Gate with Spring profile: @Profile("dev"). Remove permitAll for /h2-console/**. Set a strong database password.
CRITICAL

C4: Hardcoded Default Password for All Registrations

OWASP: A07:2021 - Identification and Authentication Failures

CWE: CWE-798 (Use of Hard-coded Credentials)

File: AuthService.java:66

Every newly registered user receives the password password123 regardless of what they enter. The registration password is completely ignored.

passwordEncoder.encode("password123") // Always used

Impact: All user accounts have the same known password. Mass account compromise.

Remediation: Use the user-supplied password: passwordEncoder.encode(dto.getPassword()). Enforce minimum password policy (8+ chars, mixed case, numbers, symbols).
CRITICAL

C5: Reports & Dashboard Endpoints - Zero Authentication

OWASP: A01:2021 - Broken Access Control

CWE: CWE-306 (Missing Authentication for Critical Function)

Files: ReportController.java:19-42, DashboardController.java:16-19

All report endpoints (dashboard stats, case reports, activity logs) and the dashboard endpoint have no authentication or authorization checks. Any unauthenticated user can access all case data and activity logs.

Impact: Unauthenticated access to all case reports, dashboard statistics, and activity logs.

Remediation: Add requireAuth(request) and appropriate role checks to every endpoint in ReportController and DashboardController.
CRITICAL

C6: Password Change - No Ownership Check + Hardcoded Bypass

OWASP: A01:2021 - Broken Access Control

CWE: CWE-625 (Permissive Password Match) + CWE-862 (Missing Authorization)

File: UserController.java:71-80

Any authenticated user can change any other user's password by passing their ID. If oldPassword is not provided, it defaults to "admin123".

if (oldPassword == null) oldPassword = "admin123"; // HARDCODED BYPASS

Impact: Any user can take over any account including admin. Complete privilege escalation.

Remediation: Verify user.getId().equals(id) or require admin role. Remove hardcoded default password. Always require the old password.

5. High Severity Findings

HIGH

H1: No Brute Force Protection on Login

CWE: CWE-307 (Improper Restriction of Excessive Authentication Attempts)

File: AuthController.java:18-21

No rate limiting, account lockout, or CAPTCHA on login endpoint. Unlimited password guessing attempts are possible. The User model has a lockedOut field but it is never used.

Remediation: Implement failed login tracking, lock accounts after 5 failures. Add rate limiting (e.g., Bucket4j). Implement CAPTCHA after 3 failures.
HIGH

H2: No JWT Token Revocation Mechanism

CWE: CWE-613 (Insufficient Session Expiration)

File: JwtUtil.java:36-43, auth.service.ts:32-37

JWTs are valid for 24 hours with no revocation capability. Stolen tokens remain valid. Logout only removes the token from localStorage client-side.

Remediation: Implement token blacklist (Redis-backed). Use short-lived access tokens (15min) + refresh tokens. Check blacklist in JwtAuthFilter.
HIGH

H3: Unrestricted File Upload - No Type Validation

CWE: CWE-434 (Unrestricted Upload of File with Dangerous Type)

Files: FileUploadController.java:60-64, EvidenceService.java:238-249

No file type whitelist, no MIME validation, no content/magic byte validation. Accepts any file type including .jsp, .war, .exe, .sh. Could lead to Remote Code Execution if uploads directory is served.

Remediation: Implement strict file type whitelist. Validate MIME type AND magic bytes. Use Files.probeContentType(). Reject executables.
HIGH

H4: All Uploads Served Without Authentication

CWE: CWE-306 (Missing Authentication for Critical Function)

File: SecurityConfig.java:40

/uploads/** is permitAll(). Evidence photos, case documents, and task documents are accessible to anyone without authentication via predictable URLs.

Remediation: Remove permitAll() for /uploads/**. Serve files through an authenticated controller with access control. Use signed URLs for temporary access.
HIGH

H5: userId Parameter Manipulation (8 Endpoints)

CWE: CWE-639 (Authorization Bypass Through User-Controlled Key)

Files: CaseController.java, TaskController.java, EvidenceController.java

8 endpoints accept userId from the request body/params instead of deriving it from the JWT. Any user can impersonate any other user. Some default to userId=1 (admin).

Remediation: Remove userId from all request inputs. Always derive from the authenticated JWT token: user.getId().
HIGH

H6: IDOR - Insecure Direct Object References on Cases

CWE: CWE-639 (Authorization Bypass Through User-Controlled Key)

File: CaseController.java:57-131

Any authenticated user can access any case by ID. No ownership or assignment verification. Same applies to case history, tasks, and evidence.

Remediation: Verify the user has a UserCaseRole for the requested case, or allow only admin/case manager access.
HIGH

H7: Missing Case Status/Close Authorization

CWE: CWE-862 (Missing Authorization)

File: CaseController.java:82-98

The case status change and close endpoints have no authentication or role checks. userId is taken from the request body.

Remediation: Add requireAuth and role checks. Use authenticated user ID from JWT.
HIGH

H8: Stored XSS via Unsanitized User Input

CWE: CWE-79 (Cross-site Scripting - Stored)

Files: CaseService.java:81-90, EvidenceService.java:82-87, FileUploadController.java:70

User-supplied fields (caseName, background, justification, fileTitle) are stored without sanitization. Angular auto-escapes {{ }} but any future API consumer or admin panel using innerHTML would be vulnerable.

Remediation: Implement server-side input sanitization. Strip HTML tags and event handlers. Use a SanitizeUtil library.
HIGH

H9: CORS Allows Credentials with Wildcard Headers

CWE: CWE-942 (Permissive Cross-domain Policy)

File: SecurityConfig.java:55

setAllowedHeaders(List.of("*")) with setAllowCredentials(true) allows any origin to send any headers including Authorization.

Remediation: Restrict to: List.of("Authorization", "Content-Type")

6. Medium Severity Findings

MEDIUM

M1: CSRF Protection Disabled

CWE: CWE-352 (Cross-Site Request Forgery)

File: SecurityConfig.java:35

CSRF is globally disabled. While JWT Bearer tokens partially mitigate this, cookie-based attacks remain possible.

Remediation: If using JWT Bearer exclusively, document the rationale. Otherwise implement CSRF tokens for state-changing operations.
MEDIUM

M2: IDOR on Tasks, Evidence, and Users

CWE: CWE-639 (Authorization Bypass Through User-Controlled Key)

Files: TaskController.java:33-43, EvidenceController.java:36-46, UserController.java:38-69

Any authenticated user can view any task, any evidence item, list all users, and view any user's details (PII exposure).

Remediation: Filter by user's assigned cases. Restrict user listing to admin roles. Verify case membership for tasks/evidence.
MEDIUM

M3: File Upload Endpoints Missing Role Checks

CWE: CWE-862 (Missing Authorization)

File: FileUploadController.java:45-137

Case and task file upload endpoints only call requireAuth() with no role checks. Any authenticated user can upload files to any case/task.

Remediation: Add role checks (canManageCases, canManageEvidence). Verify user has access to the target case/task.
MEDIUM

M4: Task Note Author ID Spoofing

CWE: CWE-639 (Authorization Bypass Through User-Controlled Key)

File: TaskController.java:116-129

The authorId for task notes is taken from the request body, allowing attribution of notes to other users.

Remediation: Always use user.getId() for the author. Never accept authorId from client.
MEDIUM

M5: No HTTPS Enforcement

CWE: CWE-319 (Cleartext Transmission of Sensitive Information)

File: application.properties:1

No SSL/TLS configuration. JWT tokens transmitted in plaintext over HTTP.

Remediation: Configure SSL or terminate TLS at a reverse proxy/load balancer. Enforce HSTS.
MEDIUM

M6: Error Messages Leak Implementation Details

CWE: CWE-209 (Generation of Error Message Containing Sensitive Information)

Files: FileUploadController.java:90, EvidenceService.java:273

Raw IO exception messages returned to clients, revealing file system paths and OS information.

Remediation: Return generic error messages. Log detailed exceptions server-side only.
MEDIUM

M7: Open Registration Without Rate Limiting

CWE: CWE-799 (Improper Control of Interaction Frequency)

File: SecurityConfig.java:38, AuthController.java:24-27

/api/auth/register is permitAll() with no input validation, rate limiting, or email verification. Allows mass account creation.

Remediation: Add input validation (DTO annotations), email verification, rate limiting.
