1. System Overview

JENE-DFCMS follows a strict chain-of-custody workflow where each role has defined responsibilities. The system enforces RBAC at both frontend (UI visibility) and backend (API enforcement) levels.

Core Principle

  • Admin sets up the system (users, teams, settings)
  • Case Manager receives requests, creates cases, assigns tasks, monitors progress
  • Investigator performs forensic work, uploads evidence and findings
  • QA reviews investigation quality before delivery
  • Authoriser approves/rejects case closure and authorization
  • Requester submits investigation requests and receives final reports

3. Status Workflows

Case Status Flow

CREATED ➔ PENDING ➔ OPEN ➔ CLOSED or REJECTED or ARCHIVED
  • CREATED - Initial state when case is first created by Case Manager
  • PENDING - Awaiting authorization or additional information
  • OPEN - Active investigation in progress
  • CLOSED - Investigation complete, authorized by Authoriser
  • REJECTED - Case rejected by Authoriser (returns to OPEN)
  • ARCHIVED - Long-term storage after closure

Task Status Flow

CREATED ➔ QUEUED ➔ ALLOCATED ➔ PROGRESS ➔ QA ➔ DELIVERY ➔ COMPLETE
  • CREATED - Task defined by Case Manager
  • QUEUED - Awaiting investigator assignment
  • ALLOCATED - Assigned to specific investigator
  • PROGRESS - Forensic examination in progress
  • QA - Submitted for quality assurance review
  • DELIVERY - QA approved, delivering to Case Manager
  • COMPLETE - Task finished and accepted

Evidence Status Flow

INACTIVE ➔ ACTIVE ➔ ARCHIVED or DESTROYED

4. Detailed Role Responsibilities

⚙ ADMIN

Purpose: System administrator. Sets up and maintains the entire system.

  • Create/manage departments and teams
  • Create user accounts with roles
  • Enable/disable user accounts
  • Configure system settings (retention, formats)
  • View all cases, tasks, evidence across system
  • Change case status (OPEN, CLOSED, ARCHIVED)
  • Override any RBAC restriction
  • Monitor system-wide activity via Dashboard
  • Upload case/task documents
  • Upload final forensic reports
  • Delete final reports

📄 CASE MANAGER

Purpose: Manages the lifecycle of forensic cases from creation to closure.

  • Create new cases (set name, classification, type, priority)
  • Edit case details (background, location, justification)
  • Upload case documents (COC, letters, forms, memos)
  • Check/uncheck document checklist items
  • Create tasks within cases
  • Assign tasks to investigators
  • Monitor task progress and status
  • Upload evidence and evidence documents
  • Upload final forensic reports
  • Change case status (OPEN, PENDING)
  • Close cases (with reason)
  • Link/unlink related cases
  • View case history and audit trail
  • Request authorization from Authoriser

🔎 INVESTIGATOR

Purpose: Performs the actual forensic investigation work.

  • View assigned tasks
  • Upload evidence (photos, devices, media)
  • Upload evidence documents (exam results, reports)
  • Add task notes with findings
  • Change task status (PROGRESS, COMPLETE)
  • Check in/check out evidence (chain of custody)
  • View case details for assigned cases
  • Upload documents to tasks
  • Record evidence retention dates

✓ QA (Quality Assurance)

Purpose: Reviews investigation quality before delivery.

  • View assigned tasks for QA review
  • Review task notes and findings
  • Verify evidence integrity
  • Approve or return tasks for revision
  • Add QA comments to tasks
  • Change task status to QA/DELIVERY
  • View case details

✅ AUTHORISER

Purpose: Authorizes or rejects case closure and final reports.

  • Review case findings and evidence
  • Authorize case closure
  • Reject case (with reason) - returns to OPEN
  • View all case details, tasks, evidence
  • View case history and audit trail
  • Cannot create cases or tasks
  • Cannot modify evidence

✉ REQUESTER

Purpose: Submits investigation requests and receives final reports.

  • Create new cases (investigation requests)
  • Provide case background and justification
  • View own case details
  • Download final forensic reports
  • Cannot create tasks
  • Cannot upload evidence
  • Cannot modify other users' cases

6. Step-by-Step Scenario

Here is a realistic scenario showing how a corporate data breach investigation flows through the system:

Phase 1: System Setup (One-time)

  1. Admin logs in with admin/admin123
  2. Admin creates departments: Forensics, Cybercrime, Operations
  3. Admin creates teams: Digital Forensics, Cybercrime Unit, Intelligence
  4. Admin creates user accounts for each team member with appropriate roles
  5. Admin configures system settings (retention period, date format, etc.)

Phase 2: Case Initiation

  1. Requester (or Case Manager) submits an investigation request with background info
  2. Case Manager reviews the request and creates a formal case:
    • Sets classification (Confidential, Secret, etc.)
    • Sets type (Incident Response, Fraud Investigation, etc.)
    • Sets priority (Critical, High, Normal, Low)
    • Sets deadline and location
    • Uploads Chain of Custody form, agency letter, authority letter
    • Checks document checklist items as documents are received

Phase 3: Task Assignment

  1. Case Manager creates tasks for the case:
    • "Email Log Extraction" - assign to John (Investigator)
    • "Disk Forensics" - assign to Priya (Investigator)
    • "Network Traffic Analysis" - assign to Ankit (Investigator)
  2. Case Manager sets case status to OPEN

Phase 4: Investigation

  1. Investigator (John) logs in and sees assigned tasks
  2. Investigator (John) uploads evidence:
    • USB drive found at scene (photo + bag number)
    • Forensic image of server disk
  3. Investigator (John) performs examination and adds task notes:
    • "Recovered 3 email accounts from compromised server"
    • "Found C2 communication to IP 192.168.1.100"
  4. Investigator (John) uploads examination documents:
    • Email extraction report (PDF)
    • Network capture analysis (PCAP summary)
  5. Investigator (John) sets task status to COMPLETE

Phase 5: Quality Assurance

  1. QA (Mike) reviews John's completed task
  2. QA verifies evidence integrity and chain of custody
  3. QA approves the task or returns it for revision

Phase 6: Report & Closure

  1. Case Manager reviews all completed tasks
  2. Case Manager uploads the Final Forensic Investigation Report
  3. Case Manager requests authorization from Authoriser
  4. Authoriser (David) reviews the case, findings, and final report
  5. Authoriser AUTHORIZES the case closure
  6. Case Manager sets case status to CLOSED
  7. Requester receives notification and downloads the final report

8. Security Enforcement Rules

  • JWT Authentication: All API endpoints (except login) require a valid JWT token with HMAC-SHA256 signature verification
  • Token Expiry: JWT tokens expire after 1 hour; users must re-authenticate
  • Brute Force Protection: Account locks after 5 failed login attempts
  • Password Policy: Minimum 8 characters, BCrypt cost 12
  • Input Sanitization: XSS prevention at service layer (script tags, event handlers stripped)
  • File Upload Security: Whitelist of 30+ safe extensions, blocklist of 15+ dangerous extensions
  • IDOR Prevention: userId derived from JWT token, not request body
  • CORS: Restricted to localhost:4200 only, limited headers
  • Security Headers: CSP, HSTS, X-Content-Type-Options, X-Frame-Options: DENY
  • Frontend RBAC: Navbar hides/shows links based on selected role; guards protect routes
  • Backend RBAC: RoleHelper checks roles at every controller endpoint; unauthorized returns HTTP 403
