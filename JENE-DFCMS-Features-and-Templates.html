1System Overview & Architecture

JENE-DFCMS is a full-stack Digital Forensic Case Management System built with Angular 17 (frontend) and Spring Boot 3.2.0 (backend), designed for managing forensic investigations, evidence, and chain of custody.

Technology Stack

LayerTechnologyDetails
FrontendAngular 17TypeScript, NgModule-based, Material-style CSS
BackendSpring Boot 3.2.0Java 17, Maven, REST API
DatabaseH2 (file-based)JPA/Hibernate, auto-schema update
AuthenticationJWT (JSON Web Tokens)HMAC-SHA, 24h expiry, BCrypt passwords
SecuritySpring Security 6Stateless sessions, CORS, role-based access
BuildMavenSpring Boot starter parent 3.2.0

System Statistics

27 Entity Classes 6 Service Classes 7 Controllers 33 API Endpoints 20 Repositories 16 Frontend Components 6 Frontend Services 6 User Roles 27 Database Tables 8 DTOs

Architecture Pattern

MVC + Service Layer: Controller → Service → Repository → Database. JWT filter sits before all authenticated endpoints. RoleHelper component provides centralized authorization checks.

2Authentication & Security Features

2.1 JWT Authentication

Login

POST /api/auth/login with username/password. Returns JWT token, username, roles, and userId. Token stored in localStorage.

Self-Registration

POST /api/auth/register. Creates account with default role INVESTIGATOR and password "password123".

Token Interceptor

Angular HttpInterceptor automatically attaches Bearer token to all API requests. Auto-logout on 401 response.

Token Expiry

JWT tokens expire after 24 hours (86,400,000 ms). Users must re-authenticate after expiry.

2.2 Password Security

  • BCrypt Hashing: All passwords are hashed with BCrypt before storage. No plain-text passwords in database.
  • Password Change: PUT /api/users/{id}/password with old and new password. Validates old password before changing.
  • Account Lockout: Configurable number of failed login attempts before account lockout (default: 5).
  • Active/Inactive: Users can be deactivated without deletion.

12.2 Audit Trail Features

  • Automatic Recording: All changes to cases, tasks, and evidence are automatically recorded
  • User Attribution: Each change records which user made it and when
  • Snapshot Data: History records capture a snapshot of the entity at the time of change

  • Immutable: History records cannot be modified or deleted
  • Chronological Display: History is displayed newest-first in the UI

18Workflow Templates

18.1 Complete Case Lifecycle

End-to-End Case Workflow

  1. REQUESTER creates case → status: DRAFT
  2. CASE_MANAGER reviews and opens case → status: OPEN
  3. CASE_MANAGER creates tasks (e.g., Disk Forensics, Email Extraction)
  4. CASE_MANAGER assigns INVESTIGATOR to tasks
  5. CASE_MANAGER assigns QA reviewer to tasks
  6. INVESTIGATOR works on tasks, adds forensic notes
  7. INVESTIGATOR creates evidence items, checks in/out evidence
  8. CASE_MANAGER starts case work → status: IN_PROGRESS
  9. INVESTIGATOR/QA review tasks, mark as complete
  10. CASE_MANAGER requests authorization → status: PENDING_AUTH
  11. AUTHORISER reviews and approves → status: AUTHORIZED
  12. CASE_MANAGER closes case → status: CLOSED

18.2 Task Assignment Workflow

Task Assignment Process

  1. CASE_MANAGER creates task under case
  2. CASE_MANAGER assigns primary INVESTIGATOR (principle=true)
  3. CASE_MANAGER optionally assigns secondary INVESTIGATOR (principle=false)
  4. CASE_MANAGER assigns primary QA reviewer
  5. CASE_MANAGER optionally assigns secondary QA reviewer
  6. INVESTIGATOR begins work, changes status to IN_PROGRESS
  7. INVESTIGATOR adds forensic notes during work
  8. INVESTIGATOR completes work, requests QA review
  9. QA reviews work, adds review notes
  10. QA passes or fails QA review
  11. If passed: task moves to COMPLETED
  12. If failed: task returns to IN_PROGRESS for rework

18.3 Evidence Handling Workflow

Evidence Collection & Analysis Process

  1. INVESTIGATOR collects evidence at scene
  2. INVESTIGATOR creates evidence record in system
  3. INVESTIGATOR checks OUT evidence (removes from storage)
  4. INVESTIGATOR performs forensic analysis
  5. INVESTIGATOR adds forensic notes with findings
  6. INVESTIGATOR checks IN evidence (returns to storage)
  7. Chain of custody record is created for each transfer
  8. Evidence status updated: INACTIVE → ACTIVE → IN_ANALYSIS → ANALYZED

18.4 Case Authorization Workflow

Authorization Process

  1. CASE_MANAGER completes all tasks
  2. CASE_MANAGER changes status to PENDING_AUTH
  3. AUTHORISER receives notification (manual)
  4. AUTHORISER reviews case report and evidence
  5. If approved: status → AUTHORIZED
  6. If rejected: status → REJECTED (with reason)
  7. If rejected: CASE_MANAGER can reopen case
